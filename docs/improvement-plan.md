# 개선 기획서

## 변경 이력

| 날짜 | 변경 내용 | 이유 |
|---|---|---|
| 2026-03-31 | 프로젝트 전반 검토 결과를 바탕으로 개선 기획서 초안 작성 | 보안, 데이터 분리, 운영 안정성, 품질 관리 우선순위 정리 |

## 목적

현재 프로젝트는 귀여운 다이어리형 태스크 매니저로서 UI 완성도와 기본 CRUD 흐름은 잘 갖춰져 있다.  
반면 인증이 추가된 이후에도 데이터 계층이 사용자 단위로 분리되지 않았고, 운영 환경에서 문제가 될 수 있는 인증/스키마/오류 처리 리스크가 남아 있다.

이 문서의 목적은 다음 3가지를 정리하는 것이다.

1. 무엇을 먼저 고쳐야 하는지 우선순위를 명확히 한다.
2. 각 개선 항목의 범위, 기대 효과, 작업 순서를 정의한다.
3. 구현 전에 팀이 합의해야 할 정책 포인트를 분리한다.

## 현재 진단 요약

### 1. 보안 및 권한 경계

- 로그인은 JWT 쿠키 기반으로 동작하지만, `tasks` 데이터는 아직 사용자별로 분리되지 않는다.
- `JWT_SECRET` 미설정 시 프록시가 요청을 통과시키는 fail-open 동작이 있다.
- 로그인 후 이동 경로 `from` 검증이 없어 오픈 리다이렉트 가능성이 있다.
- API가 내부 오류 메시지를 그대로 클라이언트에 반환한다.

### 2. 데이터 및 운영 구조

- API 요청 중에 `CREATE TABLE`, `ALTER TABLE`을 수행하고 있다.
- 스키마 변경이 마이그레이션으로 분리되어 있지 않아 운영 안정성이 낮다.
- 태스크 정렬/필터링 로직은 작동하지만, 다계정 구조로 확장할 준비는 아직 되지 않았다.

### 3. 품질 및 유지보수성

- 자동화 테스트가 없다.
- 한글 문자열과 문서 일부가 인코딩 깨짐 상태로 남아 있다.
- 사용자 메시지, 메타데이터, 문서 품질이 실제 제품 인상에 영향을 줄 수 있다.

## 우선순위

### P0. 즉시 처리해야 하는 항목

#### P0-1. 사용자별 태스크 분리

- 목표: 로그인한 사용자 본인의 태스크만 조회/생성/수정/삭제되도록 변경
- 핵심 작업:
  - `tasks` 테이블에 `user_id` 컬럼 추가
  - 기존 API에서 JWT의 `userId`를 읽어 쿼리 조건에 반영
  - `PATCH`, `DELETE` 시 `id`뿐 아니라 `user_id`까지 함께 검사
  - 기존 데이터 마이그레이션 정책 수립
- 기대 효과:
  - 가장 큰 권한 누수 리스크 제거
  - 향후 개인화 기능 확장 기반 확보

#### P0-2. 인증 fail-open 제거

- 목표: `JWT_SECRET`이 없으면 보호 경로가 열리지 않도록 변경
- 핵심 작업:
  - 프록시에서 secret 미설정 시 요청 통과 금지
  - 개발 환경에서도 명시적 오류를 반환하거나 로그인 페이지로 유도
  - 서버 시작 시 환경 변수 검증 로직 도입 검토
- 기대 효과:
  - 잘못된 배포 설정이 곧바로 보안 구멍이 되는 문제 차단

#### P0-3. 로그인 리다이렉트 검증

- 목표: 로그인 후 내부 경로로만 이동되도록 제한
- 핵심 작업:
  - `from` 값이 `/`로 시작하는 상대 경로인지 검증
  - 외부 URL, 프로토콜 포함 값, 이중 슬래시 경로 차단
- 기대 효과:
  - 피싱 악용 가능성 축소

### P1. 다음 스프린트에서 처리할 항목

#### P1-1. 마이그레이션 체계 도입

- 목표: 런타임 DDL 제거
- 핵심 작업:
  - 스키마 생성/변경을 별도 마이그레이션 스크립트로 분리
  - 배포 전/후 실행 절차 문서화
  - API에서 `CREATE TABLE`, `ALTER TABLE` 제거
- 기대 효과:
  - 운영 안정성 향상
  - 장애 원인 추적 용이

#### P1-2. 오류 응답 및 로깅 정책 정리

- 목표: 사용자에게는 안전한 메시지, 서버에는 충분한 진단 정보 제공
- 핵심 작업:
  - 클라이언트 응답에서 raw `error.message` 노출 제거
  - 공통 API 에러 포맷 정의
  - 서버 로그 레벨 및 포맷 정리
- 기대 효과:
  - 정보 노출 감소
  - 운영 대응 속도 개선

#### P1-3. 입력 검증 강화

- 목표: API가 프런트엔드에만 의존하지 않도록 방어선 추가
- 핵심 작업:
  - `text`, `priority`, `category`, `due_date` 서버 검증
  - 허용값 화이트리스트 적용
  - 빈 문자열, 비정상 날짜, 과도한 길이 제한
- 기대 효과:
  - 데이터 무결성 향상
  - 예상치 못한 클라이언트 호출에도 안정성 유지

### P2. 품질 개선 및 제품 완성도 향상

#### P2-1. 인코딩 정리

- 목표: UI와 문서의 한글 깨짐 제거
- 핵심 작업:
  - 소스 파일과 문서를 UTF-8로 통일
  - 깨진 문자열 복원
  - 에디터/저장소 인코딩 규칙 명시
- 기대 효과:
  - 사용자 경험 개선
  - 문서 신뢰도 회복

#### P2-2. 테스트 도입

- 목표: 핵심 회귀를 자동으로 잡을 수 있게 만들기
- 추천 범위:
  - 인증 API 테스트
  - 태스크 API 권한 테스트
  - 카테고리 정규화 유닛 테스트
  - 로그인 후 리다이렉트 처리 테스트
- 기대 효과:
  - 인증/권한 회귀 방지
  - 리팩터링 안전성 향상

#### P2-3. UX 세부 개선

- 목표: 로그인/오류/빈 상태 경험 개선
- 핵심 작업:
  - 인증 만료 시 명확한 안내 메시지 제공
  - 네트워크 오류와 권한 오류 구분
  - 로그아웃 UI 및 현재 사용자 표시 검토
- 기대 효과:
  - 실제 사용성 개선
  - 문제 상황에서 혼란 감소

### 3단계. 품질 상향

## 구현 시 고려사항

### 데이터 마이그레이션

- 기존 `tasks` 데이터가 이미 존재할 수 있으므로 일괄 소유자 지정 정책이 필요하다.
- 선택지:
  - 기존 데이터를 `admin` 계정으로 귀속
  - 기존 데이터를 보관용 테이블로 이동
  - 초기화 후 새 정책으로 재시작

### 인증 구조

- 현재 커스텀 JWT 구조는 유지 가능하다.
- 다만 사용자 수가 늘거나 OAuth, 가입, 비밀번호 재설정이 필요해지면 인증 라이브러리 도입 여부를 다시 검토할 수 있다.

### 문서 업데이트

- 실제 구현이 시작되면 아래 문서도 함께 갱신해야 한다.
- `docs/auth-system.md`: 인증 로직 및 권한 범위 변경 반영
- `docs/services.md`: 마이그레이션 도구, 테스트 도구 등 신규 패키지 반영
- 필요 시 `docs/data-model.md` 신설: `users`-`tasks` 관계와 마이그레이션 정책 정리

## 완료 기준

다음 조건을 만족하면 1차 개선이 완료된 것으로 본다.

1. 서로 다른 두 계정이 로그인해도 상대 태스크가 보이지 않는다.
2. `JWT_SECRET`이 없을 때 보호 경로 접근이 차단된다.
3. 로그인 후 외부 URL로 리다이렉트되지 않는다.
4. API가 내부 오류 문자열을 직접 노출하지 않는다.
5. 스키마 변경이 요청 처리 중 실행되지 않는다.
6. 최소한의 인증/권한 테스트가 CI 또는 로컬에서 반복 가능하다.

## 결론

이 프로젝트의 가장 중요한 다음 단계는 "로그인 기능이 있다"는 상태에서 "사용자별로 안전하게 분리된 서비스" 상태로 넘어가는 것이다.  
디자인과 기본 기능은 이미 충분히 매력적이므로, 이제는 보안 경계와 운영 신뢰성을 먼저 끌어올리는 것이 전체 제품 가치를 가장 크게 높이는 방향이다.
